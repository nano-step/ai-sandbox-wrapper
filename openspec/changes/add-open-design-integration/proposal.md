## Why

Users want AI coding agents (like OpenCode) running inside ai-sandbox-wrapper containers to drive [`nexu-io/open-design`](https://github.com/nexu-io/open-design) ‚Äî an open-source, agent-native Claude Design alternative ‚Äî to programmatically generate design artifacts (HTML prototypes, decks, landing pages, dashboards).

OpenCode is a better prompt engineer than a human user: it can iterate, refine briefs, parse SSE responses, and chain multi-turn design refinements. But today there's no path for an OpenCode container to reach an open-design daemon without manual Docker juggling: figuring out networks, ports, tokens, container DNS ‚Äî friction that defeats the sandbox-wrapper's "single command, just works" philosophy.

open-design's daemon was explicitly designed for external callers (bearer-token auth, SSE streaming, project isolation via `project_id`) ‚Äî but the integration with ai-sandbox-wrapper does not exist yet. This change adds a first-class `open-design` tool to the sandbox so any sandboxed agent (OpenCode today, others later) can reach the daemon via `http://ai-open-design:7456` with a pre-injected `$OD_API_TOKEN`, zero manual setup, and no port exposure to the host by default.

## What Changes

### New tool installer

- Add `lib/install-open-design.sh` following the existing `install-{tool}.sh` pattern used by all other tools (opencode, claude, gemini, etc.)
- Add `dockerfiles/open-design/Dockerfile` based on the official `docker.io/vanjayak/open-design:latest` image with daemon-mode environment defaults (`OD_BIND_HOST=0.0.0.0`)
- Register `open-design` in `setup.sh` tool selection menu and in CI (`.gitlab-ci.yml`)

### Lifecycle subcommands (long-running daemon, not ephemeral CLI)

Unlike all existing tools (which are ephemeral CLIs invoked once per session), open-design is a long-running HTTP daemon. `bin/ai-run` gains a new dispatch path when tool=open-design:

- `ai-run open-design init` ‚Äî one-time: generate persistent API token, write to `~/.ai-sandbox/env` as `OD_API_TOKEN`, create Docker network `ai-sandbox` if missing, create volume `ai-open-design-data`
- `ai-run open-design start [--expose]` ‚Äî boot daemon container in detached mode on network `ai-sandbox`; only publish port 7456 to host when `--expose` is passed
- `ai-run open-design stop` ‚Äî `docker stop ai-open-design` (data in volume preserved)
- `ai-run open-design status` ‚Äî show running state, port, token presence, daemon health (`curl /api/health`)
- `ai-run open-design logs [-f]` ‚Äî `docker logs [-f] ai-open-design`
- `ai-run open-design restart` ‚Äî stop then start with same args

### Network + token plumbing for OpenCode containers

- `ai-run` ensures every container (existing tools included) joins the shared Docker network `ai-sandbox` so service discovery via container name `ai-open-design` works
- Auto-inject `OD_DAEMON_URL=http://ai-open-design:7456` and `OD_API_TOKEN=<from ~/.ai-sandbox/env>` into every spawned container that opts in (default: opencode, claude, codex, gemini, opt-out via flag)
- Mount `~/.ai-sandbox/env` (already exists for API keys) into containers so token is read at shell-start time
- Mount volume `ai-open-design-data` **read-only** into agent containers at `/workspace/.od` so agents can read artifacts/HTML files generated by open-design

### Helper command inside agent containers

- Ship `od-status`, `od-health` shell wrappers (installed into `/usr/local/bin` of base image) so an agent shell can do quick connectivity checks against the daemon ‚Äî useful when the daemon was started after the agent container, helping the agent verify connectivity without exporting env vars manually

### Documentation

- Update `README.md` (or new `docs/open-design.md`) with the standard workflow: `ai-run open-design init` (once) ‚Üí `ai-run open-design start` ‚Üí `ai-run opencode` (auto-connects)

## Impact

### Affected specs

- **New**: `open-design-integration` spec capturing daemon lifecycle, networking contract, token persistence, and per-agent injection rules
- **Modified**: `base-image` (add `od-status`/`od-health` shell wrappers), `container-runtime` (Docker network membership rule for all tools)

### Affected code

- New: `lib/install-open-design.sh`, `dockerfiles/open-design/Dockerfile`, `dockerfiles/open-design/od-status`, `dockerfiles/open-design/od-health`
- Modified: `bin/ai-run` (new lifecycle dispatch when first arg = `open-design`; network join logic; env injection logic), `setup.sh` (menu entry), `.gitlab-ci.yml` (CI build), `dockerfiles/base/Dockerfile` (add `od-status`/`od-health` helpers)

### Compatibility

- **Backward compatible**: Existing `ai-run opencode`, `ai-run claude`, etc. continue to work identically. Network membership is additive (containers without open-design daemon running simply have unreachable `$OD_DAEMON_URL`, which is harmless).
- **No breaking changes** to existing config files (`~/.ai-sandbox/config.json`, `~/.ai-sandbox/env`) ‚Äî only adds `OD_API_TOKEN` line and creates a new tool directory `~/.ai-sandbox/tools/open-design/`

### Security

- Default deny: daemon listens only inside Docker network `ai-sandbox`, not on host. `--expose` is opt-in.
- Bearer token (256-bit random) auto-generated on first `init`, persisted in `~/.ai-sandbox/env` (same trust boundary as user API keys)
- Volume `ai-open-design-data` mounted **read-only** into agent containers ‚Äî agent cannot corrupt design state, only read outputs
- No SSRF risk: open-design has built-in internal-IP/SSRF blocking at daemon edge (per upstream's `/api/proxy/*` endpoints)

### Resource cost

- Daemon container: ~384 MB RAM, ~3-5s startup
- Persisted volume: grows with project history (SQLite + HTML artifacts)
- Image: ~500 MB pulled once

### Out of scope (deferred)

- MCP server wrapper around open-design API (Phase 2 ‚Äî current change uses raw HTTP via `curl` from inside agent shells)
- Mode B (1-1 pairing per OpenCode session) ‚Äî current change ships Mode A only (one shared daemon serves multiple OpenCode sessions, each as a separate `project_id`)
- Auto-discovery / `od-connect` interactive helper (current change relies on Dockerfile-level env vars; helper can come later if friction is observed)
