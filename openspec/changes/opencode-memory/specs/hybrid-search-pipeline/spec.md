## ADDED Requirements

### Requirement: BM25 Full-Text Search

The system SHALL provide BM25 full-text search via SQLite FTS5 with porter stemming and unicode61 tokenization. Search SHALL complete in under 50ms for typical queries against indexes with up to 10,000 documents.

#### Scenario: Fast keyword search

- **WHEN** a user calls `memory_search` with query "authentication flow" on an index containing 5,000 documents
- **THEN** the system SHALL return ranked results within 50ms using BM25 scoring from the FTS5 index

#### Scenario: Porter stemming normalization

- **WHEN** a user searches for "running" using BM25 search
- **THEN** the system SHALL match documents containing "run", "runs", "running", and "ran" due to porter stemming

### Requirement: Vector Semantic Search

The system SHALL provide vector semantic search using sqlite-vec with cosine similarity distance metric. Embeddings SHALL be 384-dimensional float vectors generated by EmbeddingGemma-300M.

#### Scenario: Semantic similarity matching

- **WHEN** a user calls `memory_vsearch` with query "user login process"
- **THEN** the system SHALL return documents about "authentication flow" and "sign-in mechanism" even if they don't contain the exact query terms

#### Scenario: Cosine similarity scoring

- **WHEN** vector search returns results
- **THEN** each result SHALL include a cosine similarity score normalized to 0-1 range where 1 is most similar

### Requirement: Hybrid Query with RRF Fusion

The system SHALL implement hybrid search combining BM25 and vector results using Reciprocal Rank Fusion (RRF) with k=60. The original query SHALL receive 2× weight compared to expanded query variants.

#### Scenario: Query expansion with LLM variants

- **WHEN** a user calls `memory_query` with query "how to implement auth"
- **THEN** the system SHALL generate 2 expanded query variants using Qwen3-1.7B and execute 3 parallel searches (original + 2 variants) for both BM25 and vector

#### Scenario: RRF fusion with weighted original query

- **WHEN** combining results from 3 BM25 searches and 3 vector searches
- **THEN** the system SHALL apply RRF formula `score = sum(1/(k + rank))` with k=60 and SHALL count the original query results twice for 2× weight

#### Scenario: Top-rank bonus for exact matches

- **WHEN** a document appears in rank 1 of the original query BM25 results
- **THEN** the system SHALL apply a top-rank bonus to preserve exact keyword matches in final ranking

### Requirement: LLM Reranking

The system SHALL rerank the top 30 candidates from RRF fusion using Qwen3-Reranker-0.6B cross-encoder model. Reranking scores SHALL be normalized to 0-1 range.

#### Scenario: Cross-encoder reranking of top candidates

- **WHEN** RRF fusion produces 30 candidate results
- **THEN** the system SHALL pass each candidate with the original query to Qwen3-Reranker and SHALL receive relevance scores between 0 and 1

#### Scenario: Reranker score normalization

- **WHEN** the reranker returns raw logits
- **THEN** the system SHALL normalize scores to 0-1 range using sigmoid or min-max scaling

### Requirement: Position-Aware Score Blending

The system SHALL blend RRF scores with reranker scores using position-aware weighting to preserve exact matches. Final score SHALL be `(1 - position_weight) × rrf_score + position_weight × reranker_score` where position_weight increases with rank.

#### Scenario: Preserve top-ranked exact matches

- **WHEN** a document ranks #1 in RRF fusion with high BM25 score but receives low reranker score
- **THEN** the system SHALL apply low position_weight (high RRF influence) to preserve the exact match in top results

#### Scenario: Increase reranker influence for lower ranks

- **WHEN** a document ranks #20 in RRF fusion
- **THEN** the system SHALL apply higher position_weight to allow reranker to promote semantically relevant results

### Requirement: Score Normalization

The system SHALL normalize all scores to 0-1 range: BM25 rank-based normalization, vector cosine similarity to 0-1, and reranker scores to 0-1.

#### Scenario: BM25 rank normalization

- **WHEN** BM25 returns results with ranks 1, 2, 3, ..., N
- **THEN** the system SHALL convert ranks to scores using `1 / (k + rank)` formula

#### Scenario: Vector distance to similarity

- **WHEN** sqlite-vec returns cosine distances
- **THEN** the system SHALL convert distances to similarity scores in 0-1 range where 1 is most similar

### Requirement: Collection Filtering

The system SHALL support filtering search results to specific collections via optional collection parameter. When specified, only documents from that collection SHALL be returned.

#### Scenario: Search within sessions collection only

- **WHEN** a user calls `memory_query` with collection="sessions"
- **THEN** the system SHALL exclude documents from "memory" and other collections from search results

#### Scenario: Cross-collection search by default

- **WHEN** a user calls `memory_query` without specifying collection
- **THEN** the system SHALL search across all active collections

### Requirement: Minimum Score Threshold

The system SHALL support optional minimum score threshold filtering. Results with final blended scores below the threshold SHALL be excluded from output.

#### Scenario: Filter low-relevance results

- **WHEN** a user calls `memory_query` with minScore=0.3
- **THEN** the system SHALL exclude all results with final blended score below 0.3

#### Scenario: No threshold by default

- **WHEN** a user calls `memory_query` without minScore parameter
- **THEN** the system SHALL return all ranked results regardless of score

### Requirement: Result Format

The system SHALL return results with snippet (matched text), path (file path), line range (start and end lines), score (final blended score), and docid (document ID).

#### Scenario: Complete result metadata

- **WHEN** search returns a matching document
- **THEN** each result SHALL include snippet, path, lineStart, lineEnd, score, and docid fields

#### Scenario: Snippet extraction from matched chunk

- **WHEN** a document chunk matches the query
- **THEN** the snippet SHALL contain the relevant text portion with up to 300 characters of context
